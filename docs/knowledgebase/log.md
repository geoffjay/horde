The `horde aap-mock` fixture is the end-to-end integration target.

* **Fix**: Made the AAP host handshake tolerate diagnostic `log` frames (and unknown-type frames) emitted before `ready` — `readReadyFrame` in `internal/server/aaphost.go` skips them instead of failing the spawn with "expected ready or error, got log". Surfaced by manual QA driving the real pi-aap adapter with a `model:` set (pi emits a pre-ready log noting the model id was not in its registry). Also reused the handshake's buffered reader for the read loop so a frame buffered immediately after `ready` is not dropped. Regression test: `TestAAPHostSession_HandshakeSkipsPreReadyLog`.
* **Fix**: Stopped the TUI freezing on AAP invoke. Two issues, surfaced by manual QA invoking real pi with `auto_approve: false`: (1) `internal/app` called the blocking `Client.Invoke` (which waits on the server's SSE headers) directly in the bubbletea `Update` loop, freezing the whole UI — including `ctrl+c` — until the turn produced output; it now opens the stream in a command goroutine (`invokeStartedMsg` → `handleInvokeStarted`). (2) `internal/api` `invokeAAPAgent` didn't flush the SSE preamble, so headers weren't sent until the first event; with a turn that waits for approval before emitting anything, the client never got headers. It now flushes immediately after `WriteHeader` (verified: ~0.5ms to headers on a turn blocked on approval). Known remaining rough edges for the invoke round-out (Phase-3.6, not the approval feature): `runAAPTurn` buffers frames and only streams them to the primary client at turn end (no live streaming), and the turn is tied to the HTTP request context so leaving the invoke view cancels it.
* **Fix**: Reworked AAP invoke streaming + turn lifetime (`internal/server/aapinvoke.go`), the round-out follow-up to the freeze fix. `runAAPTurn` is now a pure turn-driver that appends frames to the invocation buffer and records the terminal result via `inv.finish(err)`; a single `streamAAPInvocation` reader serves every client (primary and reconnecting) by subscribing to the buffer and tailing live with a per-reader cursor — replacing the primary's stream-at-end path, `replayAAPInvocation`, and the `lastStreamedID` hack. The turn runs on a request-independent context (`context.Background()`), so a client disconnecting — or, in the TUI, leaving the invoke view to approve a tool call — no longer cancels it; only server/agent shutdown does. This makes the in-TUI `a`/`d` approval flow work with no TUI change (invoke → esc to the agent view → decide → the turn completes) and streams tokens live. Verified by unit tests (`TestAAPInvoke_StreamsBeforeTurnCompletes`, `TestAAPInvoke_TurnSurvivesClientDisconnect`) and end-to-end (a token streamed while the approval was pending; the turn survived a client disconnect). Remaining follow-ups: precise Last-Event-ID resume (SSE id ↔ buffer id), invocation-registry eviction, an explicit turn-cancel action, and optionally rendering `a`/`d` in the invoke view itself.
