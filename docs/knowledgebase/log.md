Brought the dormant in-process `EventBus` (`internal/server/eventbus.go`) to life as a cluster-wide activity feed. The node publishes agent lifecycle events on the bus — `agent.spawned` (after a subprocess starts and its context is initialized, ADK + AAP), `agent.exiting` (on `StopAgent`), `agent.exited` (after the proc is reaped) — carrying only an agent id, origin node id, and agent name (no context payload, so no cross-node redaction needed). `GET /api/v1/events/stream` serves the bus over SSE (`streamEvents`; live-only, no backlog replay). The receiver is master-only (a slave rejects with 404) and the master never forwards, so there is no echo loop. Redefined the bus (all-events `Subscribe`, lock-held drop-on-full `Publish`, removed the dead invocation-id filter + `NewInvocationID`) and replaced the unused `EventBus()` getter with `SubscribeEvents`/`PublishClusterEvent`. * **Feature**: [Phase 4 — Distributed](/docs/knowledgebase/plans/phase-4-distributed.md) slice 5 — **gossip discovery** (completes Phase 4). Added a third `cluster.discovery_mechanism`, `gossip`: a slave finds its master through a `github.com/hashicorp/memberlist` (SWIM) ring instead of a static `server.leader` or DNS. Every node joins the ring; the master advertises `role=master` + its HTTP address in memberlist node metadata (`nodeMeta`, JSON, ≤512 B), and a slave's `gossipDiscoverer` reads the ring for the master's address, then registers/heartbeats over HTTP as before (the `leaderClient` re-resolves through the `Discoverer` each reconnect, so gossip needs no register-path change). `internal/server/gossip.go` wraps memberlist: `newGossipNode` binds listeners (fatal from `Start`), does an initial `Join(seeds)`, and runs a background re-join loop while no master is visible; `leaderAPIAddr()` scans for `role=master`; `shutdown()` (Leave+Shutdown) runs on `ctx` cancel (there is no `Stop()`). A `gossipMembers` seam keeps `gossipDiscoverer` unit-testable without binding ports. Config: `cluster.gossip_bind_addr`/`gossip_advertise_addr`/`gossip_seeds` (seeds a comma-separated **string**, not a list, so `HORDE_CLUSTER_GOSSIP_SEEDS` works without a viper slice-env hook); `Validate` requires seeds for a gossip *slave*; under `gossip` the master must set `cluster.advertise_addr` so its gossiped HTTP address is routable. **Discovery only — no leader election/failover** (out of scope; captured in the new [cluster leader failover](/docs/knowledgebase/concepts/cluster-failover.md) concept doc for a later plan). Tests: real two-node memberlist convergence on loopback (`gossip_test.go`), the discoverer via a fake, `newDiscoverer` gossip/nil cases, config validation. Verified end-to-end on a three-node docker cluster (`docker/docker-compose.gossip.yml`): slaves registered with the master having discovered its address through gossip alone (no `server.leader`), then a slave-hosted agent was invoked through the master. Docs: `docs/environment.md` (3 keys), the [Phase 4 plan](/docs/knowledgebase/plans/phase-4-distributed.md) (marked complete), the failover concept doc, roadmap, and KB index.
