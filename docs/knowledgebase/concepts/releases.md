Releases are driven by [goreleaser](https://goreleaser.com) via
`.goreleaser.yml` and triggered by `.github/workflows/release.yml` on `v*`
tag pushes. The process is fully automated: tag → build → GitHub release →
Homebrew formula.

# Release process

1. **Ensure `main` is clean and up to date:**
   ```
   git checkout main && git pull
   ```

2. **Run the full check locally:**
   ```
   task
   ```
   This runs tidy → fmt → vet → test → build. All must pass.

3. **(Optional) Verify the release build without publishing:**
   ```
   task snapshot
   ```
   Runs `goreleaser release --snapshot --clean` — builds all targets,
   archives, and checksums locally, but does not publish to GitHub or
   push the Homebrew formula. Use this to catch build issues before
   tagging.

4. **Tag and push:**
   ```
   git tag v0.X.0
   git push origin v0.X.0
   ```
   The `v*` tag push triggers `.github/workflows/release.yml`, which runs
   goreleaser to build (linux + darwin, amd64 + arm64), create the GitHub
   release with an auto-generated changelog, and push the Homebrew formula
   to `geoffjay/homebrew-tap`.

5. **Monitor the workflow:**
   ```
   gh run watch
   ```
   The release workflow appears under the "Release" action. If it fails,
   the tag can be re-pushed or the workflow can be re-run via
   `workflow_dispatch` with the tag name.

# Versioning

horde uses [semantic versioning](https://semver.org). The first release
was `v0.1.0`. While the major version is `0`, breaking changes are
expected between minor versions.

The version is injected at build time via:
```
-ldflags "-X github.com/geoffjay/horde/cmd.version={{.Version}}"
```
For plain `go build` (no ldflags), `cmd.version` defaults to `"dev"`.
`horde --version` reports the injected version; this exists primarily
so the Homebrew formula's `test do` block can verify the install.

# Changelog

The changelog is auto-generated by goreleaser from GitHub commit messages
(see `.goreleaser.yml` → `changelog`). These commit prefixes are excluded:

- `docs:` — documentation-only changes
- `test:` — test-only changes
- `chore:` — chores
- `Merge pull request` / `Merge branch` — merge commits

Use conventional commit prefixes (`feat:`, `fix:`, `refactor:`, etc.) so
the changelog is useful.

# Homebrew tap

The `brews:` block in `.goreleaser.yml` pushes a formula to
`geoffjay/homebrew-tap` on every release. Requirements:

- The `HOMEBREW_TAP_TOKEN` repo secret (a PAT with write access to
  `geoffjay/homebrew-tap`) must be set. If it's unset, the job logs a
  notice and skips the tap push — the GitHub release still succeeds.
- The formula's `test do` block runs `horde --version` and asserts the
  output contains "horde".

The `brews:` (formulas) key is deprecated in goreleaser v2.10+ in favour
of `homebrew_casks`, but casks are for macOS GUI apps and don't support
Linux. horde targets linux + darwin as a CLI, so `brews:` is correct;
`goreleaser check` reports a deprecation warning but `goreleaser release`
still works. Re-evaluate if a future goreleaser major removes `brews:`.

# Files involved

| File | Purpose |
| --- | --- |
| `.goreleaser.yml` | Build targets, archives, checksums, changelog filters, Homebrew formula |
| `.github/workflows/release.yml` | CI trigger: `v*` tag push or manual `workflow_dispatch` |
| `Taskfile.yml` | `task snapshot` (local, no publish) and `task release` (local, publishes) |
| `cmd/version` | The `--version` flag handler; value injected via ldflags |

# Local release (advanced)

`task release` runs `goreleaser release --clean` locally. This publishes
to GitHub and pushes the Homebrew formula. It requires `GITHUB_TOKEN`
and `HOMEBREW_TAP_TOKEN` in the environment. Prefer the tag-push
workflow for normal releases; `task release` is for debugging or
when CI is unavailable.

# Manual rerun

If the release workflow fails, it can be re-run via:
```
gh workflow run release.yml -f tag=v0.X.0
```
The tag must already exist and point at the desired commit. goreleaser
uses the checked-out tag, not the branch HEAD.
